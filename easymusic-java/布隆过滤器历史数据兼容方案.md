# 布隆过滤器历史数据兼容方案设计文档

## 一、方案概述

### 1.1 背景
系统引入布隆过滤器后，需要保证布隆过滤器、Redis缓存、MySQL数据库三者的数据一致性，特别是历史数据（音乐、用户、订单）需要补齐到布隆过滤器中，防止缓存穿透。

### 1.2 设计原则（只有一条核心思路）
- **只在夜间低峰期做一致性校准**：白天不做大批量同步，只在每天凌晨利用线程池+分页慢慢跑任务，把历史数据补齐到布隆过滤器，同时校正 Redis 与 MySQL 的一致性。

换句话说：**不在系统启动时一次性加载历史数据，只在每天凌晨定时扫一遍，把 Bloom + Redis + MySQL 对齐。**

### 1.3 技术选型
- **线程池**：Java `ThreadPoolExecutor`，夜间批量异步处理数据
- **定时任务**：Spring `@Scheduled`，每天凌晨触发一次校准任务
- **存储**：
  - MySQL：权威数据源（music/user/order 表）
  - Redis：业务已有缓存
  - BloomFilter：基于 Redisson 的分布式布隆过滤器组件

## 二、核心功能：夜间一致性校准任务

### 2.1 执行时机
- **执行时间**：每天凌晨 2:00（可以根据实际低峰期调整）
- **执行频率**：每天执行一次
- **触发方式**：Spring `@Scheduled(cron = "0 0 2 * * ?")`

### 2.2 整体流程

目标：在夜间慢慢扫表，把 **历史数据补齐到布隆过滤器**，同时顺带把 **Redis 缓存缺失的热点数据补上**。

高层步骤：
```
1. 按类型分三类跑：音乐 / 用户 / 订单
2. 每类数据都按主键分页从 MySQL 扫描（pageNo 从 1 开始，pageSize 固定）
3. 对每一页的数据，用线程池并发处理
4. 每条数据的处理逻辑：
   - 如果 Bloom 中不存在，则补一条（add）
   - 如果 Redis 中没有缓存，但 MySQL 有数据，可按需重建缓存
5. 整个表扫完后，输出一次统计日志即可
```

### 2.3 线程池 & 分页参数（建议值）

| 参数项           | 建议值      | 说明                     |
|------------------|-------------|--------------------------|
| 线程池核心线程数 | 4~8         | 视机器 CPU 核心数调整    |
| 线程池最大线程数 | 8~16        | 避免过高压垮 MySQL/Redis |
| 队列容量         | 100         | 控制排队任务数           |
| 分页大小         | 1000 条/页  | 每次从 MySQL 拉取的数据  |
| 批次大小         | 100 条/批   | 每批提交给线程池的数量   |

> 说明：参数只是建议，真正上线时可以根据机器负载和表数据量做调优。

### 2.4 三类数据的一致性逻辑

统一思路：**以 MySQL 为准，把 Bloom 填满，把 Redis 补齐（可选）。**

1. **音乐数据（music_info）**
   - 从 `music_info` 分页查询 `music_id`
   - 对每个 `music_id`：
     - 调用 `bloomFilterComponent.mightContainMusic(musicId)` 判断
     - 如果为 `false`，调用 `bloomFilterComponent.addMusic(musicId)` 补上
     - 如需缓存：检查 Redis 中是否已有 `MusicInfo` 缓存，如无可从 DB 查一次并写入

2. **用户数据（user_info）**
   - 从 `user_info` 分页查询 `user_id`
   - 类似音乐逻辑，使用 `mightContainUser / addUser`

3. **订单数据（pay_order_info）**
   - 从 `pay_order_info` 分页查询 `order_id`
   - 类似音乐逻辑，使用 `mightContainOrder / addOrder`

> 这里 **不会** 去清理 Bloom 中“多出来”的数据（布隆过滤器允许误判，但不允许漏判），所以只补、不删。

### 2.5 处理策略（伪代码）

以音乐数据为例，伪代码如下（用户/订单同理）：

```java
int pageNo = 1;
int pageSize = 1000;

while (true) {
    List<String> musicIdList = musicInfoMapper.selectMusicIdsByPage(pageNo, pageSize);
    if (musicIdList == null || musicIdList.isEmpty()) {
        break;
    }

    // 按批次丢给线程池处理
    int batchSize = 100;
    for (int i = 0; i < musicIdList.size(); i += batchSize) {
        int end = Math.min(i + batchSize, musicIdList.size());
        List<String> batch = musicIdList.subList(i, end);

        executor.execute(() -> {
            for (String musicId : batch) {
                // 1. 补布隆过滤器
                if (!bloomFilterComponent.mightContainMusic(musicId)) {
                    bloomFilterComponent.addMusic(musicId);
                }

                // 2. 可选：补 Redis 缓存（视业务需要打开）
                // if (!redisComponent.existsMusicCache(musicId)) {
                //     MusicInfo musicInfo = musicInfoMapper.selectByMusicId(musicId);
                //     if (musicInfo != null) {
                //         redisComponent.setMusicInfo(musicInfo);
                //     }
                // }
            }
        });
    }

    pageNo++;
}
```

### 4.2 线程池配置

```java
@Configuration
public class ThreadPoolConfig {

    @Bean("bloomFilterLoaderExecutor")
    public ThreadPoolExecutor bloomFilterLoaderExecutor() {
        return new ThreadPoolExecutor(
            5,                              // 核心线程数
            10,                             // 最大线程数
            60L,                            // 空闲线程存活时间
            TimeUnit.SECONDS,               // 时间单位
            new LinkedBlockingQueue<>(100), // 等待队列
            new ThreadFactoryBuilder()
                .setNameFormat("bloom-filter-loader-%d")
                .build(),
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
    }
}
```

## 五、注意事项

- 建议只在 **夜间低峰期** 跑任务，避免影响白天线上请求。
- 分页大小、线程池大小需要结合实际数据量和机器配置做压测后再确定。
- 夜间任务如果失败，可以第二天重新跑一遍，Bloom 只会「多加不减」，不会影响正确性。

## 六、预期效果

- 每天凌晨自动把历史数据 **补齐到布隆过滤器**，避免缓存穿透。
- Redis 中该有但没的热点数据，会在夜间被逐步补充（如果打开补缓存逻辑）。
- 白天只用现有查询链路（布隆 → Redis → MySQL），不做大批量校准操作。

## 六、注意事项

### 6.1 配置要求
- Spring Boot主类需要添加 `@EnableScheduling` 注解
- 确保Redis连接正常
- 确保数据库连接正常
- 确保布隆过滤器已初始化

### 6.2 性能考虑
- 历史数据加载采用异步方式，避免阻塞应用启动
- 控制加载速度，避免对数据库造成压力（使用Thread.sleep）
- 使用线程池控制并发数，避免资源耗尽
- 分页查询避免内存溢出

### 6.3 容错处理
- 异常情况记录日志

## 七、预期效果

1. **历史数据加载**：
   - 系统启动后异步完成历史数据加载
   - 加载过程对业务无影响

2. **数据一致性**：
   - 每天夜间自动校验数据一致性
   - 自动修复不一致的数据
   - 保证布隆过滤器、Redis、MySQL三者一致
